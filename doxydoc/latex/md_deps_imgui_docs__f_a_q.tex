{\bfseries{TL;DR\+: Please try to refer to this library as \char`\"{}\+Dear Im\+Gui\char`\"{}.}}

The library started its life as \char`\"{}\+Im\+Gui\char`\"{} due to the fact that I didn\textquotesingle{}t give it a proper name when when I released 1.\+0, and had no particular expectation that it would take off. However, the term I\+M\+G\+UI (immediate-\/mode graphical user interface) was coined before and is being used in variety of other situations (e.\+g. Unity uses it own implementation of the I\+M\+G\+UI paradigm). To reduce the ambiguity without affecting existing code bases, I have decided on an alternate, longer name \char`\"{}\+Dear Im\+Gui\char`\"{} that people can use to refer to this specific library.

Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} is about making things that are simple, efficient and powerful.

Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} is built to be efficient and scalable toward the needs for A\+A\+A-\/quality applications running all day. Somehow. You can alter the look of the interface to some degree\+: changing colors, sizes, padding, rounding, fonts. Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} takes advantage of a few C++ languages features for convenience but nothing anywhere Boost insanity/quagmire. Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} does N\+OT require C++11 so it can be used with most old C++ compilers. Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} doesn\textquotesingle{}t use any C++ header file. Language-\/wise, function overloading and default parameters are used to make the A\+PI easier to use and code more terse. Doing so I believe the A\+PI is sitting on a sweet spot and giving up on those features would make the A\+PI more cumbersome. Other features such as namespace, constructors and templates (in the case of the Im\+Vector$<$$>$ class) are also relied on as a convenience.

There is an auto-\/generated \href{https://github.com/cimgui/cimgui}{\texttt{ c-\/api for Dear Im\+Gui (cimgui)}} by Sonoro1234 and Stephan Dilly. It is designed for creating binding to other languages. If possible, I would suggest using your target language functionalities to try replicating the function overloading and default parameters used in C++ else the A\+PI may be harder to use. Also see \href{https://github.com/ocornut/imgui/wiki/Bindings}{\texttt{ Bindings}} for various third-\/party bindings.

\hypertarget{md_deps_imgui_docs__f_a_q_autotoc_md16}{}\doxysubsubsection{Q\+: How can I tell whether to dispatch mouse/keyboard to Dear Im\+Gui or to my application?}\label{md_deps_imgui_docs__f_a_q_autotoc_md16}
You can read the {\ttfamily io.\+Want\+Capture\+Mouse}, {\ttfamily io.\+Want\+Capture\+Keyboard} and {\ttfamily io.\+Want\+Text\+Input} flags from the \mbox{\hyperlink{struct_im_gui_i_o}{Im\+Gui\+IO}} structure.

e.\+g. {\ttfamily if (\mbox{\hyperlink{namespace_im_gui_a3179e560812f878f3961ce803a5d9302}{Im\+Gui\+::\+Get\+I\+O()}}.Want\+Capture\+Mouse) \{ ... \}}


\begin{DoxyItemize}
\item When {\ttfamily io.\+Want\+Capture\+Mouse} is set, imgui wants to use your mouse state, and you may want to discard/hide the inputs from the rest of your application.
\item When {\ttfamily io.\+Want\+Capture\+Keyboard} is set, imgui wants to use your keyboard state, and you may want to discard/hide the inputs from the rest of your application.
\item When {\ttfamily io.\+Want\+Text\+Input} is set to may want to notify your OS to popup an on-\/screen keyboard, if available (e.\+g. on a mobile phone, or console OS).
\end{DoxyItemize}

{\bfseries{Note\+:}} You should always pass your mouse/keyboard inputs to Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}}, even when the io.\+Want\+Capture\+X\+XX flag are set false. This is because imgui needs to detect that you clicked in the void to unfocus its own windows.

{\bfseries{Note\+:}} The {\ttfamily io.\+Want\+Capture\+Mouse} is more accurate that any manual attempt to \char`\"{}check if the mouse is hovering a window\char`\"{} (don\textquotesingle{}t do that!). It handle mouse dragging correctly (both dragging that started over your application or over an imgui window) and handle e.\+g. modal windows blocking inputs. Those flags are updated by {\ttfamily \mbox{\hyperlink{namespace_im_gui_ab3f1fc018f903b7ad79fd10663375774}{Im\+Gui\+::\+New\+Frame()}}}. Preferably read the flags after calling \mbox{\hyperlink{namespace_im_gui_ab3f1fc018f903b7ad79fd10663375774}{New\+Frame()}} if you can afford it, but reading them before is also perfectly fine, as the bool toggle fairly rarely. If you have on a touch device, you might find use for an early call to {\ttfamily \mbox{\hyperlink{namespace_im_gui_a8156f9d63c3184d40e7e303be175b5fb}{Update\+Hovered\+Window\+And\+Capture\+Flags()}}}.

{\bfseries{Note\+:}} Text input widget releases focus on \char`\"{}\+Return Key\+Down\char`\"{}, so the subsequent \char`\"{}\+Return Key\+Up\char`\"{} event that your application receive will typically have {\ttfamily io.\+Want\+Capture\+Keyboard == false}. Depending on your application logic it may or not be inconvenient. You might want to track which key-\/downs were targeted for Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}}, e.\+g. with an array of bool, and filter out the corresponding key-\/ups.)

\hypertarget{md_deps_imgui_docs__f_a_q_autotoc_md25}{}\doxysubsubsection{Q\+: Why are multiple widgets reacting when I interact with a single one? $<$br$>$\+Q\+: How can I have multiple widgets with the same label or with an empty label?}\label{md_deps_imgui_docs__f_a_q_autotoc_md25}
A primer on labels and the ID Stack...

Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} internally need to uniquely identify UI elements. Elements that are typically not clickable (such as calls to the Text functions) don\textquotesingle{}t need an ID. Interactive widgets (such as calls to Button buttons) need a unique ID. Unique ID are used internally to track active widgets and occasionally associate state to widgets. Unique ID are implicitly built from the hash of multiple elements that identify the \char`\"{}path\char`\"{} to the UI element.


\begin{DoxyItemize}
\item Unique ID are often derived from a string label\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"OK"});          \textcolor{comment}{// Label = "OK",     ID = hash of (..., "OK")}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"Cancel"});      \textcolor{comment}{// Label = "Cancel", ID = hash of (..., "Cancel")}}
\end{DoxyCode}

\item ID are uniquely scoped within windows, tree nodes, etc. which all pushes to the ID stack. Having two buttons labeled \char`\"{}\+O\+K\char`\"{} in different windows or different tree locations is fine. We used \char`\"{}...\char`\"{} above to signify whatever was already pushed to the ID stack previously\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a581e58db0bc930bafa4a5d23093a2b99}{Begin}}(\textcolor{stringliteral}{"MyWindow"});}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"OK"});          \textcolor{comment}{// Label = "OK",     ID = hash of ("MyWindow", "OK")}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a5479d93794a004c67ceb6d13f37c8254}{End}}();}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a581e58db0bc930bafa4a5d23093a2b99}{Begin}}(\textcolor{stringliteral}{"MyOtherWindow"});}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"OK"});          \textcolor{comment}{// Label = "OK",     ID = hash of ("MyOtherWindow", "OK")}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a5479d93794a004c67ceb6d13f37c8254}{End}}();}
\end{DoxyCode}

\item If you have a same ID twice in the same location, you\textquotesingle{}ll have a conflict\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"OK"});}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"OK"});          \textcolor{comment}{// ID collision! Interacting with either button will trigger the first one.}}
\end{DoxyCode}
 Fear not! this is easy to solve and there are many ways to solve it!
\item Solving ID conflict in a simple/local context\+: When passing a label you can optionally specify extra ID information within string itself. Use \char`\"{}\#\#\char`\"{} to pass a complement to the ID that won\textquotesingle{}t be visible to the end-\/user. This helps solving the simple collision cases when you know e.\+g. at compilation time which items are going to be created\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a581e58db0bc930bafa4a5d23093a2b99}{Begin}}(\textcolor{stringliteral}{"MyWindow"});}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"Play"});        \textcolor{comment}{// Label = "Play",   ID = hash of ("MyWindow", "Play")}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"Play\#\#foo1"});  \textcolor{comment}{// Label = "Play",   ID = hash of ("MyWindow", "Play\#\#foo1")  // Different from above}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a38094c568ce398db5a3abb9d3ac92030}{Button}}(\textcolor{stringliteral}{"Play\#\#foo2"});  \textcolor{comment}{// Label = "Play",   ID = hash of ("MyWindow", "Play\#\#foo2")  // Different from above}}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a5479d93794a004c67ceb6d13f37c8254}{End}}();}
\end{DoxyCode}

\item If you want to completely hide the label, but still need an ID\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{namespace_im_gui_a57d73c1d0ef807fef734d91024092027}{Checkbox}}(\textcolor{stringliteral}{"\#\#On"}, \&\mbox{\hyperlink{glcorearb_8h_a6eba317e3cf44d6d26c04a5a8f197dcb}{b}});  \textcolor{comment}{// Label = "",       ID = hash of (..., "\#\#On")   // No visible label, just a checkbox!}}
\end{DoxyCode}

\item Occasionally/rarely you might want change a label while preserving a constant ID. This allows you to animate labels. For example you may want to include varying information in a window title bar, but windows are uniquely identified by their ID. Use \char`\"{}\#\#\#\char`\"{} to pass a label that isn\textquotesingle{}t part of ID\+:

\hypertarget{md_deps_imgui_docs__f_a_q_autotoc_md27}{}\doxysubsubsection{Q\+: How can I display an image? What is Im\+Texture\+I\+D, how does it work?}\label{md_deps_imgui_docs__f_a_q_autotoc_md27}
Short explanation\+:
\begin{DoxyItemize}
\item Please read Wiki entry for examples\+: \href{https://github.com/ocornut/imgui/wiki/Image-Loading-and-Displaying-Examples}{\texttt{ https\+://github.\+com/ocornut/imgui/wiki/\+Image-\/\+Loading-\/and-\/\+Displaying-\/\+Examples}}
\item You may use functions such as {\ttfamily \mbox{\hyperlink{namespace_im_gui_ad8705ac3b718c1b2e84e7b8c34f90249}{Im\+Gui\+::\+Image()}}}, {\ttfamily \mbox{\hyperlink{namespace_im_gui_ac79ef64e8b9e1db73d8237f6a999da14}{Im\+Gui\+::\+Image\+Button()}}} or lower-\/level {\ttfamily \mbox{\hyperlink{struct_im_draw_list_ac1f529d53d7ba118484e47e49218c603}{Im\+Draw\+List\+::\+Add\+Image()}}} to emit draw calls that will use your own textures.
\item Actual textures are identified in a way that is up to the user/engine. Those identifiers are stored and passed as Im\+Texture\+ID (void$\ast$) value.
\item Loading image files from the disk and turning them into a texture is not within the scope of Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} (for a good reason).
\end{DoxyItemize}

{\bfseries{Please read documentations or tutorials on your graphics A\+PI to understand how to display textures on the screen before moving onward.}}

Long explanation\+:
\begin{DoxyItemize}
\item Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}}\textquotesingle{}s job is to create \char`\"{}meshes\char`\"{}, defined in a renderer-\/agnostic format made of draw commands and vertices. At the end of the frame those meshes (\mbox{\hyperlink{struct_im_draw_list}{Im\+Draw\+List}}) will be displayed by your rendering function. They are made up of textured polygons and the code to render them is generally fairly short (a few dozen lines). In the examples/ folder we provide functions for popular graphics A\+PI (Open\+GL, DirectX, etc.).
\item Each rendering function decides on a data type to represent \char`\"{}textures\char`\"{}. The concept of what is a \char`\"{}texture\char`\"{} is entirely tied to your underlying engine/graphics A\+PI. We carry the information to identify a \char`\"{}texture\char`\"{} in the Im\+Texture\+ID type. Im\+Texture\+ID is nothing more that a void$\ast$, aka 4/8 bytes worth of data\+: just enough to store 1 pointer or 1 integer of your choice. Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} doesn\textquotesingle{}t know or understand what you are storing in Im\+Texture\+ID, it merely pass Im\+Texture\+ID values until they reach your rendering function.
\item In the \href{https://github.com/ocornut/imgui/tree/master/examples}{\texttt{ examples/}} bindings, for each graphics A\+PI binding we decided on a type that is likely to be a good representation for specifying an image from the end-\/user perspective. This is what the {\itshape examples} rendering functions are using\+:

 For example, in the Open\+GL example binding we store raw Open\+GL texture identifier (G\+Luint) inside Im\+Texture\+ID. Whereas in the Direct\+X11 example binding we store a pointer to I\+D3\+D11\+Shader\+Resource\+View inside Im\+Texture\+ID, which is a higher-\/level structure tying together both the texture and information about its format and how to read it.
\item If you have a custom engine built over e.\+g. Open\+GL, instead of passing G\+Luint around you may decide to use a high-\/level data type to carry information about the texture as well as how to display it (shaders, etc.). The decision of what to use as Im\+Texture\+ID can always be made better knowing how your codebase is designed. If your engine has high-\/level data types for \char`\"{}textures\char`\"{} and \char`\"{}material\char`\"{} then you may want to use them. If you are starting with Open\+GL or DirectX or Vulkan and haven\textquotesingle{}t built much of a rendering engine over them, keeping the default Im\+Texture\+ID representation suggested by the example bindings is probably the best choice. (Advanced users may also decide to keep a low-\/level type in Im\+Texture\+ID, and use \mbox{\hyperlink{struct_im_draw_list}{Im\+Draw\+List}} callback and pass information to their renderer)
\end{DoxyItemize}

\hypertarget{md_deps_imgui_docs__f_a_q_autotoc_md31}{}\doxysubsubsection{Q\+: How can I interact with standard C++ types (such as std\+::string and std\+::vector)?}\label{md_deps_imgui_docs__f_a_q_autotoc_md31}

\begin{DoxyItemize}
\item Being highly portable (bindings for several languages, frameworks, programming style, obscure or older platforms/compilers), and aiming for compatibility \& performance suitable for every modern real-\/time game engines, dear imgui does not use any of std C++ types. We use raw types (e.\+g. char$\ast$ instead of std\+::string) because they adapt to more use cases.
\item To use \mbox{\hyperlink{namespace_im_gui_aa72b5760a3a206729723a587ac7c8519}{Im\+Gui\+::\+Input\+Text()}} with a std\+::string or any resizable string class, see \href{https://github.com/ocornut/imgui/blob/master/misc/cpp/imgui_stdlib.h}{\texttt{ misc/cpp/imgui\+\_\+stdlib.\+h}}.
\item To use combo boxes and list boxes with std\+::vector or any other data structure\+: the \mbox{\hyperlink{namespace_im_gui_aa895095bdc7a2907375c555e245575ea}{Begin\+Combo()}}/\+End\+Combo() A\+PI lets you iterate and submit items yourself, so does the \mbox{\hyperlink{namespace_im_gui_a4003d6a6ab57f2fb43db80c6339a1aed}{List\+Box\+Header()}}/\+List\+Box\+Footer() A\+PI. Prefer using them over the old and awkward \mbox{\hyperlink{namespace_im_gui_aa2979368da5b9e98d368449b36d166b2}{Combo()}}/\+List\+Box() api.
\item Generally for most high-\/level types you should be able to access the underlying data type. You may write your own one-\/liner wrappers to facilitate user code (tip\+: add new functions in \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}}\+:\+: namespace from your code).
\item Dear \mbox{\hyperlink{namespace_im_gui}{Im\+Gui}} applications often need to make intensive use of strings. It is expected that many of the strings you will pass to the A\+PI are raw literals (free in C/\+C++) or allocated in a manner that won\textquotesingle{}t incur a large cost on your application. Please bear in mind that using std\+::string on applications with large amount of UI may incur unsatisfactory performances. Modern implementations of std\+::string often include small-\/string optimization (which is often a local buffer) but those are not configurable and not the same across implementations.
\item If you are finding your UI traversal cost to be too large, make sure your string usage is not leading to excessive amount of heap allocations. Consider using literals, statically sized buffers and your own helper functions. A common pattern is that you will need to build lots of strings on the fly, and their maximum length can be easily be scoped ahead. One possible implementation of a helper to facilitate printf-\/style building of strings\+: \href{https://github.com/ocornut/Str}{\texttt{ https\+://github.\+com/ocornut/\+Str}} This is a small helper where you can instance strings with configurable local buffers length. Many game engines will provide similar or better string helpers.
\end{DoxyItemize}
